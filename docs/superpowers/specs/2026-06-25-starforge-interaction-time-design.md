# starforge — 介入と時間操作の設計（"眺めるだけ"からの脱却）

- 日付: 2026-06-25
- 対象: `web/starforge`（WebGPU N体重力サンドボックス）
- 状態: 設計確定 / B案（忠実度は最大・段階リリース）で実装

## 1. 課題と狙い

現状は 6 シーンを鑑賞でき、ドラッグで星追加・ズーム・パン・一時停止・リセットはあるが、
**能動的な手応え・時間操作・"事件"（衝突）が乏しく、結局ゆったり眺めるだけ**になっている。

体験の軸は「**両方バランス**」= 介入（操作）と時間操作を組み合わせた“宇宙で遊ぶ”総合体験。
時間は **前進が主役（どんどん先へ）／巻き戻しはサブ（やり直しポイント）** の非対称構成。
衝突は **シーン（題目）ごとに最適な挙動を出し分け**、**WebGPU を活かす**（GPU 側で完結、CPU 読み戻し無し）。

## 2. 全体方針（B案：忠実度は同じ・段階リリース）

到達点は「全部入り」。リスクの高い GPU 合体を最後に隔離するため 2 波に分割する。

- **第1波**: 時間操作（前進＋軽量巻き戻し）＋ 4 つの操作動詞。既存バッファの延長で組め、リスク低。これだけで“介入して時間を操る”体験が立ち上がる。
- **第2波**: GPU 衝突・合体（アトミック合体＋コンパクション）＋ シーン別プロファイル。一番深い GPU 作業をここに隔離。

非目標（YAGNI）: 連続スクラブのフルタイムライン、3D 化、サウンド、保存/共有、マルチプレイ。

## 3. 時間・進行モデル（第1波・主役）

### 3.1 速度＝サブステップ数（dt は固定）
現状の Speed スライダーは `dt` を直接スケールするだけで、leapfrog は `dt` を上げると破綻する。
**速度の正体を「1 フレームあたりの物理サブステップ数 N」に変える**。`dt` は各シーンの安定値に固定し、
`render loop` 内で `sim.step(encoder)` を N 回呼ぶ。

- レート段: 一時停止(0) / コマ送り(1 step) / 0.25x / 0.5x / 1x / 2x / 4x / 8x / 16x / 32x / 64x / **Hyperlapse**。
  - 1x = 1 substep/frame。Nx = N substeps/frame（小数倍はフレーム間引き or 単一 step）。
  - Hyperlapse は N をさらに大きく（数十〜200）。数億年相当を数秒に圧縮し、トレイル蓄積で構造形成が“見える”。
- **負荷ガード**: `N_eff = clamp(N, 0, maxSubsteps)`。`maxSubsteps` は body 数に応じて自動で下げる
  （O(n²) なので、`maxSubsteps ≈ budget / max(1, count)` のような単純式）。実効レートを HUD に表示し、
  目標に届かない時は「実効 Nx」を出して黙って間引かない。

### 3.2 エポック時計
シミュ経過時間 `simTime += dt * N_eff` を毎フレーム加算し、HUD に表示（例 `T+ 3.4 Myr`、単位は表示上のスケール）。
前進に意味が出て「時間が過ぎるのを観る」体験になる。リセット/シーン切替で 0 に戻す。

### 3.3 巻き戻し（サブ）= 軽量チェックポイント
連続スクラブは作らない。**チェックポイント方式**:

- バッファ（positions/velocities/masses）と `activeCount`・`simTime` を 1 枚として GPU 上にコピー保存
  （`encoder.copyBufferToBuffer`）。リングで最大 8 枚保持。
- 取得タイミング: 一定間隔（例 5 秒相当）＋ **ユーザ操作の直前**（破壊的介入の前に必ず 1 枚）。
- 操作: 「直前のチェックポイントへ戻す」「1 つ前へ」。連続スクラブ UI は持たない。
- 概算メモリ: 1 枚 ≈ 80KB（4096 体）× 8 ≈ 640KB。許容。

## 4. 操作動詞（第1波）

### 4.1 入力モデルとツール選択
- 左ドラッグ = アクティブツールの操作 / 右・Shift ドラッグ = パン / ホイール = ズーム（カメラは不変）。
- ツールは下部ツールバー＋数字キー 1–4 で切替。既存の「ドラッグで星追加」もツールの 1 つに整理。
- 既定ツールは「掴んで投げる」。

### 4.2 GPU ネイティブな介入（読み戻し無し）
ピッキングのための GPU→CPU 読み戻しは行わない。代わりに **ポインタ介入 uniform**
`{ worldPos: vec2, worldVel: vec2, radius: f32, strength: f32, mode: u32, active: u32 }` を用意し、
**`interact` コンパクト compute パス**を各サブステップで走らせて、半径内の body の速度に作用させる。
スポーン系は既存 `appendBody()` を使う。

- **掴んで投げる (1)**: 押下中、カーソル近傍に引力（mode=attract）。カーソル速度 `worldVel` を弱く伝播。
  離した瞬間、近傍 body に `worldVel * k` のインパルス（mode=throw・1 回）。重力グラブ＝押しっぱなしで吸い寄せ。
- **質量体を置く (2)**: クリックで大質量天体（恒星/BH）を `appendBody`。質量はサブスライダー or ホイールで調整、
  リング表示で可視化。
- **衝突を仕掛ける (3)**: ドラッグで「発射点→速度ベクトル」を引き、その軌道で body（または小クラスタ）を放つ。
  既存選択を狙点へ放つ用途にも使える。
- **爆発・乱す (4)**: クリック地点を中心に半径内へ外向きインパルス（距離減衰）。mode=shock・1 回。

### 4.3 容量管理
スポーンに備え、バッファは初期シーン数より大きい `MAX_BODIES`（現 4096）を確保済み。
`activeCount` を超えたら拒否しログ。第2波の合体で `activeCount` が減れば枠が空く。

## 5. シーン別の衝突プロファイル（第2波）

各シーンの `params` に `collisionMode` を持たせ、題目に合わせて出し分ける（GPU 側で完結）。

| シーン | 既定 collisionMode | 効果 |
|---|---|---|
| 実太陽系 | none/soft | 安定軌道を保つ |
| 銀河衝突 | scatter（潮汐散乱） | すり抜けつつ潮汐尾 |
| 降着円盤 | merge（降着） | 中心へ合体・成長 |
| 誕生（重力崩壊） | merge（降着） | 拡散ガス→構造形成が本物に |
| 重力スリングショット | scatter | 加速・散乱 |
| 球状星団 | merge（降着） | 多体凝集・GPU の本領 |

実装の核（GPU 合体）:
- 近接判定（半径 = 質量由来の見かけ半径）。グリッド/空間ハッシュで O(n) 近接、または当面 O(n²) 流用。
- アトミックな合体クレーム（二重合体を避ける）→ 質量・運動量保存で吸収（`m1+m2`, `(m1 v1 + m2 v2)/(m1+m2)`）。
- 吸収された body を無効化 → **コンパクション**で `activeCount` を詰める（prefix-sum or アトミックカウンタ）。
- 合体時に第3波的演出（フラッシュ/衝撃波）は postfx 側で任意追加。

## 6. アーキテクチャ / モジュール境界

- `simulation.js`: substep 化（`step` は既存のまま、ループは呼び出し側）、`snapshot()/restore()`、
  `applyInteraction(params)`（interact パスの dispatch）、`simTime` 保持。第2波で合体パス追加。
- 新規 `src/interaction.js`: ツール状態・ポインタ→world 変換・介入 uniform 構築・ツールバー UI。
- 新規 `src/timecontrol.js`: レート段・サブステップ計算・負荷ガード・エポック時計・チェックポイントリング。
- `main.js`: render loop を「N_eff 回 step → render」に変更。time/interaction を結線。既存スライダー UI は
  time/interaction パネルへ再編（Gravity/Trail/Bloom は残す。Speed スライダーはレート段へ置換）。
- 新規 `src/shaders/interact.wgsl`: 介入 compute（attract/throw/shock）。第2波で `collide.wgsl` 追加。

各ユニットは単一目的・明確な IF（time は「レートと時刻」、interaction は「ポインタ→GPU 介入」）で、
個別にテスト可能にする。

## 7. エラー処理・フォールバック・検証

- WebGPU 非対応は既存のフォールバック踏襲（白画面にしない）。
- 容量超過・無効レートは安全側にクランプしログ。
- 検証: `naga` で WGSL、`node --check` で JS（既存手順）。
- 体感検証は WebGPU 対応ブラウザで目視（ユーザがデバッグして指摘するループ）。
- ブラウザキャッシュ対策（DevTools "Disable cache" or 別ポート）に留意。

## 8. リリース順（実装計画の骨子）

1. 時間レート（サブステップ化）＋エポック時計＋負荷ガード。
2. チェックポイント（snapshot/restore・操作前自動取得）。
3. 操作動詞（interact.wgsl＋interaction.js＋ツールバー）。
4. （第2波）シーン別 collisionMode ＋ GPU 合体＋コンパクション。
5. （任意）合体/衝突の postfx 演出。

各段で目視確認→指摘→修正のループ。コミットは段ごとに atomic（Claude 帰属なし・ローカル identity）。
